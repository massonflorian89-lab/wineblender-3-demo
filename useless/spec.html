📐 WineBlender 3 — Spécification consolidée

Document de référence pour la phase 1 (création projet Supabase, schéma initial)
Stack : Supabase (Postgres / EU) · Front vanilla JS · PWA

1 · Contexte

WineBlender 3 est la refonte de WineBlender en logiciel de traçabilité métier (équivalent Agreo, Process2Wine). Ambition : commercialisation à 18 mois.

Modèle centré sur le lot vivant qui évolue à chaque opération de cave, avec traçabilité ascendante et descendante complète, gestion multi-tenant pour SaaS.

Co-existence WB2 / WB3 : deux backends distincts. WB2 reste sur Firebase (existant), WB3 démarre directement sur Supabase. Pas de migration progressive de Firestore vers Postgres.

2 · Numérotation des lots

Lot racine (apport vendange)

L-YYYY-JJJ-CÉPAGE-COULEUR-SEQ

Exemple : L-2026-264-SYR-R-001 = 21/09/2026 (264e jour de l'année), Syrah, Rouge, 1er lot du jour.

Évolution sans mélange (transformation)

Le lot conserve son numéro racine et reçoit un suffixe d'étape :

SuffixeÉtapeExemple
.DBDébourbéL-2026-264-SYR-R-001.DB
.FAFermentation alcooliqueL-2026-264-SYR-R-001.FA
.VVin (fin FA)L-2026-264-SYR-R-001.V
.FMLFermentation malolactiqueL-2026-264-SYR-R-001.FML
.ELÉlevageL-2026-264-SYR-R-001.EL
.FFiltréL-2026-264-SYR-R-001.F
.PMPré-miseL-2026-264-SYR-R-001.PM

Lot d'assemblage (mélange de plusieurs lots)

A-YYYY-XXX-APPELLATION-COULEUR

Exemple : A-2026-001-PICPOUL-B. Les lots parents sont conservés dans la table lineage (graphe de parenté).

Apport multi-cépages : à valider avec les pratiques Richemer. Probablement traité comme un assemblage dès l'apport (lot A-… avec parents virtuels par cépage).

5 · Intrant œnologique (type d'opération unifié)

Type d'opération générique couvrant tout ajout de produit œnologique au moût ou au vin. Remplace les anciennes opérations séparées (Collage, Acidification, Désacidification, Tanisage, Bentonite curative…).

Sous-types

Sous-typeFamille de produitsStades possibles
collagePVPP, bentonite, gélatine, caséine, charbon, protéine de pois, colle de poissonMoût · FA · Élevage · Prép. mise
acidificationAcide tartrique, acide citriqueMoût · FA · Élevage
desacidificationCarbonate de Ca, bicarbonate de KMoût · FA · Élevage
tanisageTannins œnologiquesMoût · FA · Élevage
correctionMannoprotéines, gomme arabique, CMC, métatartriqueÉlevage · Prép. mise
autreTout produit du catalogue non-classéTous

Champs (stockés en operations.donnees jsonb)

{
  "sous_type": "collage",
  "stade": "elevage",
  "produits": [
    {
      "produit_lot_id": "uuid-du-lot-commercial",
      "dose": 25,
      "unite": "g/hL"
    }
  ],
  "duree_contact_jours": 14,
  "temperature_c": 12,
  "remarques": "..."
}
Catalogue produits : table produits_catalogue (générique : famille + nom). Les lots commerciaux (avec n° lot, DLUO, fournisseur, stock) vivent dans produits_lots. Une opération référence un produit_lot_id, ce qui garantit la traçabilité fine.

6 · Schéma SQL initial (Postgres / Supabase)

Toutes les tables portent tenant_id pour multi-tenant. Voir section RLS pour l'isolation.

Tables principales

-- Multi-tenant
create table tenants (
  id uuid primary key default gen_random_uuid(),
  nom text not null,
  slug text unique not null,
  created_at timestamptz default now()
);

create table memberships (
  user_id uuid references auth.users not null,
  tenant_id uuid references tenants not null,
  role text not null default 'caviste',
  primary key (user_id, tenant_id)
);

-- Apports vendange
create table apports (
  id uuid primary key default gen_random_uuid(),
  tenant_id uuid references tenants not null,
  date_apport date not null,
  parcelle text,
  cepage text not null,
  couleur text not null check (couleur in ('R','B','RS')),
  tonnage_kg numeric,
  volume_estime_hl numeric,
  lot_id uuid,
  created_by uuid references auth.users,
  created_at timestamptz default now()
);

-- Lots (cœur du système)
create table lots (
  id uuid primary key default gen_random_uuid(),
  tenant_id uuid references tenants not null,
  numero text not null,
  millesime int not null,
  cepage text,
  couleur text,
  statut text not null check (statut in (
    'vendange','mout','mout_stabule','mout_debourbe',
    'fa','vin','fml','elevage','vin_filtre',
    'pre_mise','conditionne','sorti'
  )),
  volume_hl numeric,
  contenant_id uuid,
  lot_type text not null check (lot_type in ('apport','assemblage')),
  archived boolean default false,
  created_at timestamptz default now(),
  unique (tenant_id, numero)
);

-- Contenants (cuves, foudres, fûts, pressoirs)
create table contenants (
  id uuid primary key default gen_random_uuid(),
  tenant_id uuid references tenants not null,
  nom text not null,
  type text not null check (type in ('cuve','foudre','fut','pressoir')),
  capacite_hl numeric not null,
  localisation text,
  actif boolean default true,
  tags text[] default '{}',
  unique (tenant_id, nom)
);

-- Opérations
create table operations (
  id uuid primary key default gen_random_uuid(),
  tenant_id uuid references tenants not null,
  type text not null,
  date_op timestamptz not null,
  caviste_id uuid references auth.users,
  notes text,
  donnees jsonb,
  fiche_travail_id uuid,
  created_at timestamptz default now()
);

-- Liaison opérations ↔ lots
create table operation_lots (
  operation_id uuid references operations on delete cascade,
  lot_id uuid references lots,
  role text not null check (role in ('source','destination')),
  volume_hl numeric,
  primary key (operation_id, lot_id, role)
);

-- Catalogue produits (générique)
create table produits_catalogue (
  id uuid primary key default gen_random_uuid(),
  tenant_id uuid references tenants not null,
  famille text not null,
  nom text not null,
  unite_defaut text,
  dose_max numeric,
  actif boolean default true
);

-- Lots commerciaux de produits (avec DLUO)
create table produits_lots (
  id uuid primary key default gen_random_uuid(),
  tenant_id uuid references tenants not null,
  produit_id uuid references produits_catalogue not null,
  nom_commercial text not null,
  fournisseur text,
  numero_lot text,
  dluo date,
  stock_restant numeric,
  created_at timestamptz default now()
);

-- Liaison opération ↔ produits utilisés
create table operation_produits (
  operation_id uuid references operations on delete cascade,
  produit_lot_id uuid references produits_lots,
  dose numeric not null,
  unite text not null,
  primary key (operation_id, produit_lot_id)
);

-- Lineage (graphe de parenté entre lots)
create table lineage (
  lot_parent_id uuid references lots not null,
  lot_enfant_id uuid references lots not null,
  operation_id uuid references operations,
  primary key (lot_parent_id, lot_enfant_id)
);

-- Fiches de travail
create table fiches_travail (
  id uuid primary key default gen_random_uuid(),
  tenant_id uuid references tenants not null,
  date_fiche date not null,
  caviste_id uuid references auth.users,
  statut text default 'brouillon' check (statut in ('brouillon','validee')),
  signature_oenologue uuid references auth.users,
  signature_at timestamptz,
  notes text
);

7 · Multi-tenant & Row Level Security

Chaque table porte tenant_id. Les policies suivantes garantissent qu'un utilisateur ne voit que les données du ou des tenants auxquels il appartient (table memberships).

-- Helper : tenants de l'utilisateur courant
create or replace function user_tenants() returns setof uuid
  language sql stable security definer as $$
  select tenant_id from memberships where user_id = auth.uid();
$$;

-- À répéter sur chaque table métier
alter table apports enable row level security;

create policy "lecture tenant" on apports for select
  using (tenant_id in (select user_tenants()));

create policy "ecriture tenant" on apports for insert
  with check (tenant_id in (select user_tenants()));

create policy "modification tenant" on apports for update
  using (tenant_id in (select user_tenants()));

create policy "suppression privilegiee" on apports for delete
  using (
    exists (
      select 1 from memberships
      where user_id = auth.uid()
        and tenant_id = apports.tenant_id
        and role in ('admin','oenologue')
    )
  );
Garantie : impossible qu'un bug front fasse fuiter les données d'une cave vers une autre. La base refuse au niveau SQL, indépendamment du code applicatif.
